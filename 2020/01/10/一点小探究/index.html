一点猜测

Posted on 2020-01-10
今天碰到了两个小问题，一是ssh断连后执行的进程没有继续执行，二是刷题时遇到了python map性能下降的问题。这两个问题我从原理上剖析了一下，感觉还是有点东西可以讲的。所以记录一下。

ssh断连引发的思考

昨天在学校提供的服务器上进行cyclegan网络的训练，打算让它跑一晚上，结果第二天回到学校一看内网vpn断掉了，ssh也断连了。重新登陆以看模型精度没有进步。显然ssh断了以后模型训练的脚本也停止工作了。这是为啥呢？经过一番思考，我想给出我自己的推测。
首先先引入Linux系统中父进程，子进程，信号，信号处理程序的概念，如果想要更清楚地了解这些，请见《操作系统原理》和《深入理解计算机系统》

我们可以认为进程总是处于下面三种状态之一————运行，暂停，终止。父进程可以通过fork函数创建一个新的子进程。(fork和execve命令结合使用可以用来运行程序)子进程会得到与父进程用户级虚拟地址空间相同的(但是独立的)一份拷贝。子进程还获得与父进程任何打开文件描述符相同的拷贝。

当一个进程由于某种状态终止时，内核并不是立即把它从系统中清除，而是处于终止状态，直到它被父进程回收。假如一个进程终止了但是没被回收，我们就将它成为僵尸进程，将由init进程收养。这样将占用系统宝贵的资源。

那什么是信号呢？信号就是一条消息，通知进程某种类型的事件已经发生了，其实就是Linux里kill命令里传递的那个信号，比方说信号9的名字时SIGKILL，代表杀死程序。信号处理程序则是根据收到的信号，来做出各种处理。那么我们可以猜测了，是不是ssh底下所有的进程都是ssh的子进程呢？ssh收到信号前，给所有子进程发一个KILL信号，所有子进程的信号处理程序收到KILL信号以后做出”自杀”的动作，最后被回收呢？这样一来，我们就解释的通了。首先来验证下，ssh_shell中的进程是不是ssh的子进程。这里演示采用wsl.值得一提的是我这里用了免密登录，需要在Windows本机上生成公私钥对(应该是基于RSA算法的)，然后把本机的公钥重命名为 authorized_keys扔到服务器的对应目录下(\wsl$\Ubuntu-18.04\home\saber.ssh)

性能下降的map

今天刷leetcode，碰到了一道题我用了map，给的用例有点苛刻，好几万个用例并且有大量重复数据。按普通的算法分析来讲，我的算法是一定不可能超时的。然而最终还是超时了。会想起前一阵子我看的一个类lua自制语言的实现，我也做出了map性能下降的猜测。
哈希表的本质，就是把key通过哈希算法映射，所以list等无法哈希的对象自然也不能成为键，哈希冲突处理算法有很多，相信大家在《数据结构》里一定也学过。我们猜测采用的哈希方法是开放探测法，这涉及冲突探测链和伪删除。
下面是检索 key 的一般流程。
倘若 key 就在哈希表中，也存在该 key 的同义词，通过遍历冲突探测链上所有同义词的 slot,逐个比对其中的 key ， 一定能找到该 key 正确的 slot，找到时便是遍历结束时。 假如 key 不在哈希表中，之前又插入了该 key 的同义词，就是说也产生了冲突探测链，我们 读取冲突探测链上的 slot 肯定找不到该 key，而系统又不知道这个链有多长，因此哈希算法会继 续调用探测函数算出下一个 slot C 即 (entry)，我们经过比对，发现此 slot 是空闲的，至此遍历结束， 说明找不到该 key，系统提示找不到该 key 。
假设 slotC 的 key 是 keyC，那么在查找 keyC 时，由于 slotB 的删除导致了冲突探测链的断裂， 在沿着冲突探测链遍历到原来的 slotB 时发现是空闲的，则说明探测链结束了，这导致找不到 keyC，系统会报错： keyC 不存在。 其实 keyC 不是不存在，而是丢了 。 问题的根源在于冲突探测链上被删除的 slot 擦除得太干净了，必须得留下点“标记”让别人 知道这并不是冲突探测链的结束，后面还有同义词的 slot 呢
为解决这个问题，引入了“伪删除”，也就是在空闲的 slot 中留下标记，如果遍历到已删除但是有标记的slot，我们就继续遍历。
这样一来就解释的通了。在面对大量的哈希冲突以及删除时，哈希表的性能将会退化到$O(m)$($m:$冲突的平均长度)
